## Overview

This example shows how to build a customer support assistant that combines two kinds of context:

- **Shared knowledge** — for policies, FAQs, order data, and support rules.
- **User memory** — for customer-specific history, preferences, and past conversations.

## Architecture

![architecture-diagram](/customer-support-agent-architecture.png)

## Understanding the Data Model

Before you run anything, it helps to understand the scope of each HydraDB identifier:

- `HYDRADB_TENANT_ID` is the top-level workspace for the whole application. It contains the shared knowledge base.
- `HYDRADB_SUB_TENANT_ID` is the shared support knowledge space. Each user gets their own user memory scope, based on their `user_id`

A practical mental model looks like this:

```text
tenant: customer-support-agent
|
+-- sub-tenant: knowledge-base
|   +-- guidelines.md
|   +-- orders.json
|   +-- terms.md
|
+-- user-memory: user_a1b2c3
|   +-- seeded conversation history
|   +-- later chat exchanges
|
+-- user-memory: user_x9y8z7
    +-- that user's own conversation history
```

This separation matters because it prevents one customer's memory from leaking into another customer's session while still letting everyone share the same knowledge base.

## How The Essentials Work

The rest of this page explains the main pieces in plain English, with simplified code snippets. These snippets are meant to show the pattern clearly, not mirror every line of the example one-for-one.

### 1. Shared Knowledge And User Memory Are Stored Separately

The app keeps business knowledge and personal memory in different scopes.

```ts
const tenantId = process.env.HYDRADB_TENANT_ID!;
const knowledgeBaseId = process.env.HYDRADB_SUB_TENANT_ID!;

function getKnowledgeScope() {
  return {
    tenant_id: tenantId,
    sub_tenant_id: knowledgeBaseId,
  };
}

function getUserScope(userId: string) {
  return {
    tenant_id: tenantId,
    user_id: userId,
  };
}
```

Why this matters:

- shared support documents should be visible to all users
- personal history should only apply to one user
- separating them makes recall easier to reason about

### 2. The Browser Creates A Stable Demo User ID

To make the demo feel personalized, the app needs a stable browser-level identifier.

```ts
const STORAGE_KEY = "customer-support-user-id";

export function getOrCreateUserId() {
  const existing = localStorage.getItem(STORAGE_KEY);

  if (existing) {
    return existing;
  }

  const nextUserId = `user_${crypto.randomUUID()}`;
  localStorage.setItem(STORAGE_KEY, nextUserId);
  return nextUserId;
}
```

This ID is the anchor for everything user-specific:

- onboarding data
- recalled memories
- new saved chat exchanges

Without a stable ID, every refresh would feel like a brand-new customer.

### 3. First-Time Onboarding Seeds Useful Memory

The example improves the first chat experience by writing a starter conversation into HydraDB before the user asks anything.

```ts
async function seedUserMemory(userId: string, conversation: string) {
  await fetch("https://api.hydradb.com/memory/add", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.HYDRADB_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      tenant_id: process.env.HYDRADB_TENANT_ID,
      user_id: userId,
      content: conversation,
      infer: true,
    }),
  });
}
```

The important part is `infer: true`.

That tells HydraDB to do more than just store raw text. It can infer useful signals such as:

- customer preferences
- past problems
- recurring topics
- likely support context

So instead of starting from zero, the assistant begins with a memory baseline.

### 4. Knowledge Recall Answers "What Does The Business Know?"

Shared knowledge is usually recalled from the app's common support space.

```ts
async function recallKnowledge(query: string) {
  const response = await fetch("https://api.hydradb.com/recall/full_recall", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.HYDRADB_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      tenant_id: process.env.HYDRADB_TENANT_ID,
      sub_tenant_id: process.env.HYDRADB_SUB_TENANT_ID,
      query,
      max_results: 6,
    }),
  });

  return response.json();
}
```

Think of this as the answer to:

> What company documents, policies, and support records matter for this question?

For a refund question, this might return:

- refund policy chunks
- order records
- terms and conditions

### 5. User Memory Recall Answers "What Should We Remember About This Customer?"

User memory is recalled separately from the shared knowledge base.

```ts
async function recallUserMemory(userId: string, query: string) {
  const response = await fetch("https://api.hydradb.com/memory/recall", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.HYDRADB_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      tenant_id: process.env.HYDRADB_TENANT_ID,
      user_id: userId,
      query,
    }),
  });

  return response.json();
}
```

This is what keeps the assistant from answering every user as if they were anonymous.

For example, memory might tell the agent that:

- the customer already asked about the same order last week
- they prefer short answers
- they recently changed their address

### 6. The Chat Route Retrieves Both In Parallel

A clean server implementation retrieves both kinds of context at the same time.

```ts
async function getSupportContext(userId: string, question: string) {
  const [memory, knowledge] = await Promise.all([
    recallUserMemory(userId, question),
    recallKnowledge(question),
  ]);

  return { memory, knowledge };
}
```

This is a small detail, but it matters:

- it keeps the route faster
- it keeps the retrieval layers independent
- it makes debugging easier because each source has a clear job

### 7. Retrieved Context Is Turned Into A Prompt The Model Can Use

Models work best when retrieved context is formatted clearly instead of dumped in as raw JSON.

```ts
function buildSystemPrompt({
  memoryText,
  knowledgeText,
}: {
  memoryText: string;
  knowledgeText: string;
}) {
  return `
You are a customer support assistant.

Use the user memory for personalized context.
Use the knowledge base for policy and business facts.

User memory:
${memoryText || "No relevant user memory found."}

Knowledge base:
${knowledgeText || "No relevant knowledge found."}

If the answer is uncertain, say what is missing instead of guessing.
  `.trim();
}
```

The system prompt usually has three jobs:

- define the assistant's role
- inject retrieved context in a readable structure
- tell the model how to behave when context is missing

That last point is important for support systems. You want honest uncertainty, not confident guessing.

### 8. The Vercel AI SDK Streams The Answer Back To The UI

Once the context is assembled, the server can stream the response.

```ts
import { streamText } from "ai";

export async function POST(req: Request) {
  const { messages, userId } = await req.json();
  const latestMessage = messages.at(-1)?.content ?? "";

  const { memory, knowledge } = await getSupportContext(userId, latestMessage);
  const system = buildSystemPrompt({
    memoryText: formatMemory(memory),
    knowledgeText: formatKnowledge(knowledge),
  });

  const result = streamText({
    model: supportModel,
    system,
    messages,
  });

  return result.toUIMessageStreamResponse();
}
```

Why streaming matters:

- users get feedback immediately
- long answers feel faster
- the UI feels like a real support assistant instead of a blocking form submit

### 9. The New Exchange Is Saved Back As Memory

After the answer is complete, the app stores the new conversation so the next turn has more context.

```ts
async function saveExchange(
  userId: string,
  userMessage: string,
  assistantMessage: string,
) {
  await fetch("https://api.hydradb.com/memory/add", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.HYDRADB_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      tenant_id: process.env.HYDRADB_TENANT_ID,
      user_id: userId,
      content: `User: ${userMessage}\nAssistant: ${assistantMessage}`,
      infer: true,
    }),
  });
}
```

This is the compounding part of the system:

- question one creates context for question two
- question two creates context for question three
- over time the assistant becomes more familiar with the customer

### 10. The Side Panels Are More Than UI Decoration

The Knowledge Base and User Memories panels are useful because they expose the app's hidden state.

When the assistant sounds generic, check the panels:

- if knowledge is empty, your uploads or tenant settings are probably wrong
- if user memory is empty, onboarding or memory writes probably failed
- if both are present but the answer is still weak, the prompt or model instructions need work

This makes the example easier to learn from because you can inspect the retrieval inputs instead of guessing.

## Summary

This example is valuable because it shows a complete support loop, not just a chat box:

- HydraDB stores shared business knowledge
- HydraDB stores customer-specific memory
- the server retrieves both before every answer
- the Vercel AI SDK streams the response
- the app saves the final exchange back as memory

That is the essential pattern behind a support agent that can be both accurate and personal.
