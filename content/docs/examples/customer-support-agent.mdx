## Overview

This example shows how to build a customer support assistant that combines two kinds of context:

- **Shared knowledge** — for policies, FAQs, order data, and support rules.
- **User memory** — for customer-specific history, preferences, and past conversations.

## Architecture

![architecture-diagram](/customer-support-agent-architecture.png)

## Understanding the Data Model

Before you run anything, it helps to understand the scope of each HydraDB identifier:

- `HYDRADB_TENANT_ID` is the top-level workspace for the whole application. It contains the shared knowledge base.
- `HYDRADB_SUB_TENANT_ID` is the shared support knowledge space. Each user gets their own user memory scope, based on their `user_id`

A practical mental model looks like this:

```text
tenant: customer-support-agent
|
+-- sub-tenant: knowledge-base
|   +-- guidelines.md
|   +-- orders.json
|   +-- terms.md
|
+-- user-memory: user_a1b2c3
|   +-- seeded conversation history
|   +-- later chat exchanges
|
+-- user-memory: user_x9y8z7
    +-- that user's own conversation history
```

This separation matters because it prevents one customer's memory from leaking into another customer's session while still letting everyone share the same knowledge base.

## Understanding The Essentials

These are the five pieces that actually make the example work.

### 1. Split Knowledge From Memory

This example uses one tenant for the whole app and then separates data by sub-tenant. The shared knowledge base lives under the default `"knowledge-base"` sub-tenant, while each user gets their own sub-tenant based on their demo user ID.

```ts title="customer-support-agent/lib/hydradb.ts"
import { HydraDBClient } from "@hydra_db/node"

/** Default tenant for this demo app; override with HYDRADB_TENANT_ID. */
export const TENANT_ID =
  process.env.HYDRADB_TENANT_ID ?? "customer-support-agent"

/** Optional sub-tenant for namespacing within a tenant. */
export const SUB_TENANT_ID = process.env.HYDRADB_SUB_TENANT_ID ?? "knowledge-base"

let _client: HydraDBClient | null | undefined

/**
 * Singleton HydraDB client. Returns `null` when HYDRADB_API_KEY is not set
 * (chat still works, but recall / memory persistence are skipped).
 */
export function getHydraClient(): HydraDBClient | null {
  if (_client !== undefined) {
    return _client
  }
  const token = process.env.HYDRADB_API_KEY
  if (!token) {
    _client = null
    return _client
  }
  _client = new HydraDBClient({ token })
  return _client
}
```

### 2. Create A Stable User ID

The app does not ask the user to sign in. Instead, it creates a local demo ID once, stores it in `localStorage`, and reuses it across reloads. That is the identifier used for onboarding, recall, and saving future memory. The `isNew` flag is equally important, because it tells the app whether it should seed starter memory on first load.

```ts title="customer-support-agent/lib/user.ts"
export function getOrCreateDemoUserIdWithNewFlag(): { userId: string; isNew: boolean } {
  if (typeof window === "undefined") {
    return {
      userId: `${USER_ID_PREFIX}${nanoid(16)}`,
      isNew: true,
    }
  }

  const existing = normalizeDemoUserId(
    window.localStorage.getItem(USER_ID_STORAGE_KEY),
  )
  if (existing) {
    return {
      userId: existing,
      isNew: false,
    }
  }

  const created = `${USER_ID_PREFIX}${nanoid(16)}`
  window.localStorage.setItem(USER_ID_STORAGE_KEY, created)
  return {
    userId: created,
    isNew: true,
  }
}
```

```ts title="customer-support-agent/app/page.tsx"
    const { userId: id, isNew: isNeverUser } = getOrCreateDemoUserIdWithNewFlag()
    setUserId(id)
    setIsNew(isNeverUser)
  }, [])
```

### 3. Seed Memory On First Load

When `isNew` is true, the app calls `/api/onboard`. That route loads `data/conversation.json` and writes the bundled user/assistant exchanges into HydraDB under the current user's sub-tenant. Two details matter here: `infer: true` lets HydraDB extract useful memory from those exchanges, and `custom_instructions` tells HydraDB what to look for while doing that extraction.

```ts title="customer-support-agent/app/api/onboard/route.ts"
import { NextResponse } from "next/server"
import { normalizeDemoUserId } from "@/lib/user"
import { getHydraClient, TENANT_ID } from "@/lib/hydradb"

import conversationData from "@/data/conversation.json"

type UserAssistantPair = {
  user: string
  assistant: string
}

export async function POST(req: Request) {
  const client = getHydraClient()
  if (!client) {
    return NextResponse.json(
      {
        error: "HYDRADB_API_KEY is not configured",
        success: false,
      },
      { status: 503 },
    )
  }

  const payload = (await req.json()) as { userId?: unknown }
  const userId = normalizeDemoUserId(payload.userId)

  if (!userId) {
    return NextResponse.json(
      {
        error: "A valid userId is required",
        success: false,
      },
      { status: 400 },
    )
  }

  try {
    // Initialize user memory with conversation pairs
    // This creates the sub_tenant_id in HydraDB for this user and stores conversation history
    const conversationPairs = (conversationData.conversations ?? []) as UserAssistantPair[]
    console.log(conversationPairs)
    await client.upload.addMemory({
      tenant_id: TENANT_ID,
      sub_tenant_id: userId,
      upsert: true,
      memories: [
        {
          user_assistant_pairs: conversationPairs,
          infer: true,
          user_name: "HydraDB Customer",
          custom_instructions: "Extract user preferences, issues, and support history from conversations",
        },
      ],
    })

    return NextResponse.json({
      success: true,
      message: "User onboarded successfully with conversation memory",
      userId,
      conversationCount: conversationPairs.length,
    })
  } catch (e) {
    console.error("[api/onboard] addMemory failed:", e)
    return NextResponse.json(
      {
        error: e instanceof Error ? e.message : "Failed to onboard user",
        success: false,
      },
      { status: 502 },
    )
  }
}
```

### 4. Turn Retrieval Into Prompt Context

The model never sees raw HydraDB responses directly. The app converts retrieved chunks into readable text, labels them clearly, and always puts user memory before knowledge base context. That ordering is not accidental. It matches the support behavior the app wants: personalize first, then fill in policies and business facts from the shared knowledge base.

```ts title="customer-support-agent/lib/prompt.ts"
import type { HydraDB } from "@hydra_db/node"

function formatChunks(
  chunks: HydraDB.VectorStoreChunk[] | undefined,
  label: string,
): string {
  if (!chunks?.length) {
    return `(No ${label} results for this query.)`
  }
  return chunks
    .map((chunk, index) => {
      const title =
        chunk.source_title ?? chunk.source_id ?? `chunk-${index + 1}`
      const score =
        chunk.relevancy_score != null
          ? ` (score: ${Number(chunk.relevancy_score).toFixed(3)})`
          : ""
      return `### ${index + 1}. ${title}${score}\n${chunk.chunk_content}`
    })
    .join("\n\n")
}

export function formatRetrievalForPrompt(
  knowledge: HydraDB.RetrievalResult | undefined,
  preferences: HydraDB.RetrievalResult | undefined,
): string {
  const kb = formatChunks(knowledge?.chunks, "knowledge base")
  const mem = formatChunks(preferences?.chunks, "user memory")

  return [
    "## User memory context (from HydraDB recall preferences)",
    mem,
    "",
    "## Knowledge base context (from HydraDB full recall)",
    kb,
  ].join("\n")
}
```

### 5. Run The Full Chat Loop

Everything comes together in `/api/chat`. The route pulls the last user message out of the AI SDK payload, normalizes the user ID from either the request body or the `x-user-id` header, recalls both preferences and knowledge from HydraDB, builds the final system prompt, streams the answer with the Vercel AI SDK, and finally stores the completed user/assistant pair back into memory. If you understand this file, you understand the full behavior of the example.

```ts title="customer-support-agent/app/api/chat/route.ts"
import { normalizeDemoUserId } from "@/lib/user"
import { formatRetrievalForPrompt } from "@/lib/prompt"
import { getHydraClient, SUB_TENANT_ID, TENANT_ID } from "@/lib/hydradb"

import { streamText, convertToModelMessages, type UIMessage } from "ai"

export const maxDuration = 60

function getLastUserText(messages: UIMessage[]): string | undefined {
  for (let i = messages.length - 1; i >= 0; i--) {
    const m = messages[i]
    if (m.role !== "user") continue
    const text = m.parts
      .filter((p): p is { type: "text"; text: string } => p.type === "text")
      .map((p) => p.text)
      .join("")
    const trimmed = text.trim()
    if (trimmed) return trimmed
  }
  return undefined
}

const BASE_SYSTEM = `You are a knowledgeable customer support assistant created by HydraDB.
Use the provided HydraDB context when it helps answer the question.
Prioritize user memory context over global knowledge when both are relevant.
If context is missing or not relevant, rely on general knowledge and say so briefly.
Be clear, concise, and professional.`

export async function POST(req: Request) {
  const payload = (await req.json()) as {
    messages?: UIMessage[]
    userId?: unknown
  }
  const messages = payload.messages ?? []

  const userIdFromHeader = req.headers.get("x-user-id")
  const userId =
    normalizeDemoUserId(payload.userId) ??
    normalizeDemoUserId(userIdFromHeader)

  const effectiveSubTenantId = userId ?? (SUB_TENANT_ID || null)

  const lastUserText = getLastUserText(messages)
  const client = getHydraClient()

  let hydraSection = ""
  if (client && lastUserText) {
    try {
      const recallBody = {
        tenant_id: TENANT_ID,
        query: lastUserText,
        max_results: 8,
        alpha: 0.8,
        recency_bias: 0,
        ...(effectiveSubTenantId
          ? { sub_tenant_id: effectiveSubTenantId }
          : {}),
      }

      const preferences = await client.recall.recallPreferences(recallBody)
      const knowledge = await client.recall.fullRecall(recallBody)

      hydraSection = formatRetrievalForPrompt(knowledge, preferences)
    } catch (e) {
      console.error("[api/chat] HydraDB recall failed:", e)
      hydraSection =
        "(HydraDB recall failed; answering without retrieved context.)"
    }
  } else if (!client) {
    hydraSection =
      "(HydraDB is not configured; set HYDRADB_API_KEY for knowledge and memory.)"
  }

  const system =
    hydraSection.length > 0
      ? `${BASE_SYSTEM}\n\n${hydraSection}`
      : BASE_SYSTEM

  const modelMessages = await convertToModelMessages(messages)

  const result = streamText({
    model: "openai/gpt-4o-mini",
    system,
    messages: modelMessages,
    abortSignal: req.signal,
    onFinish: async ({ text }) => {
      if (!client || !lastUserText || !text?.trim()) return
      try {
        await client.upload.addMemory({
          tenant_id: TENANT_ID,
          upsert: true,
          ...(effectiveSubTenantId
            ? { sub_tenant_id: effectiveSubTenantId }
            : {}),
          memories: [
            {
              user_assistant_pairs: [
                { user: lastUserText, assistant: text.trim() },
              ],
              infer: true,
            },
          ],
        })
      } catch (e) {
        console.error("[api/chat] HydraDB addMemory failed:", e)
      }
    },
  })

  return result.toUIMessageStreamResponse()
}
```

## Summary

This example shows the full support-agent loop in one place. It gives the app a stable user identity, seeds useful memory for new users, keeps shared knowledge separate from personal memory, formats both into prompt-ready context, and then runs everything through a chat route that recalls, streams, and writes memory back after each reply. This combination is what makes the assistant feel grounded instead of generic. It can answer with business context, remember the customer across turns, and improve over time because every completed exchange becomes part of the next one.
